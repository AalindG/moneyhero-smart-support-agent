# MoneyHero Backend — Claude Orchestration Rules

## Project Overview

Node.js + Express backend for MoneyHero Smart Support Agent.
LangChain.js RAG pipeline using HNSWLib + Ollama (llama3.2).
SQLite for persistence. SSE streaming on /api/chat.
Runs in Docker via docker-compose.yml (3 services: backend, frontend, ollama).

## Agent Roster

### RAG Engineer

LangChain.js and RAG pipeline specialist. Builds document ingestion and conversational AI.
Owns vector store creation, intent classification, and agent logic.

### Principal Backend Developer

API and database layer ownership. Implements Express routes, controllers, models, and SQLite.
Integrates RAG agent as a black box. Makes architectural decisions for the API layer.

### Senior QA Engineer

Integration testing, API contract validation, error handling verification.
Writes test suites, validates Docker setup, ensures production readiness.

## Agent Routing Rules

### Parallel Dispatch

Use when tasks touch different files with no dependencies:

- rag-engineer (agent.js / ingest.js) + principal-backend-dev (controllers / models / routes)
- Scaffolder can run alongside docs generation

### Sequential Dispatch

Standard workflow:

1. **scaffolder** → sets up project structure
2. **rag-engineer** + **principal-backend-dev** → implement in parallel
3. **senior-qa-engineer** → validates, tests, integrates

**Critical dependencies:**

- Scaffold before implementation
- Docs must exist before ingest.js runs
- Both implementations complete before QA testing
- `agent.js` must export `chat()` before controllers can use it

### Background Dispatch

Use background processes for:

- npm install
- npm run ingest (slow — 30–60 seconds)
- docker compose up
- Any read-only exploration

## Critical Constraints

- NEVER hardcode URLs or model names — always use process.env
- LLM stack: `@langchain/anthropic` (ChatAnthropic) for Claude primary + Ollama fallback — both are used
- NEVER have two agents edit the same file in parallel
- Principal backend dev treats `agent.chat()` as a black box
- RAG engineer never touches Express, SQLite, routes, or controllers
- Senior QA engineer validates but doesn't rewrite core logic
- `vectorstore/` is generated by ingest.js — never manually created or committed
- `data/` holds SQLite DB — must be volume-mounted in Docker, never committed
- All code must use ES module syntax (import/export, not require)

## Docker Setup

- **Services**: backend (3001) + frontend (3000) + ollama (11434)
- **Volumes**: `vectorstore/`, `data/`, `docs/` mounted from host into backend container
- `OLLAMA_BASE_URL` is automatically `http://ollama:11434` inside Docker
- First-time setup: `./scripts/setup.sh` (pulls models, runs ingestion)
- Daily use: `docker compose up -d`

## API Endpoints

```
POST /api/session                              → { sessionId }
POST /api/chat                                 → SSE stream: data:{"token":"..."} ... data:[DONE]
POST /api/escalate                             → { success, ticketId }  (format: TKT-YYYYMMDD-NNN)
GET  /api/history/:sessionId                   → { messages: [...] }
GET  /health                                   → health check

# Admin (protected by Bearer token)
POST /api/admin/login                          → { token }  (8-hour TTL)
GET  /api/admin/sessions                       → { sessions: [...] }
GET  /api/admin/sessions/:sessionId/messages   → { sessionId, messages: [...] }
GET  /api/admin/top-questions?limit=10         → { questions: [...] }
```

## Intent Classification

- `answer`    → run RAG retrieval pipeline, stream response
- `escalate`  → return handoff message, log to escalations table
- `off_topic` → politely redirect to financial product questions

## RAG Pipeline Layers (in priority order)

1. Escalation keyword check → human handoff
2. Off-topic keyword check → redirect
3. Financial keyword check → skip classifier, go straight to RAG
4. LLM intent classifier (llama3.2:1b)
5. Catalog shortcut → reads source file directly (100% accurate)
6. Loan shortcut → builds context from all personal-loans/ docs
7. Comparison shortcut → scores card/loan bullets by keyword match
8. Product routing → specific product mentioned → read that product's doc
9. Vector fallback → semantic search with adaptive score threshold

## Error Handling

- 200: success with data
- 400: `{ error: "message" }` for client errors
- 404: session not found
- 429: rate limit or escalation cooldown exceeded
- 500: `{ error: "message" }` for server errors
- Never crash on unhandled errors
